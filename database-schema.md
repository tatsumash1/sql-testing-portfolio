# Database Schema

База данных Northwind моделирует работу торговой компании.

Основная бизнес-логика строится вокруг клиентов, заказов, товаров, поставщиков, сотрудников и служб доставки.

## Основные таблицы

### Customers

Таблица хранит информацию о клиентах компании.

Ключевые поля:

- `CustomerID` - уникальный идентификатор клиента;
- `CompanyName` - название компании клиента;
- `ContactName` - имя контактного лица;
- `ContactTitle` - должность контактного лица;
- `City` - город;
- `Country` - страна;
- `Phone` - телефон.

Связи:

- `Customers.CustomerID` связан с `Orders.CustomerID`.

QA-смысл:

- каждый заказ должен быть связан с существующим клиентом;
- данные клиента не должны быть пустыми в критичных полях;
- клиенты без заказов могут быть отдельным объектом анализа.

### Orders

Таблица хранит информацию о заказах клиентов.

Ключевые поля:

- `OrderID` - уникальный идентификатор заказа;
- `CustomerID` - клиент, который сделал заказ;
- `EmployeeID` - сотрудник, который обработал заказ;
- `OrderDate` - дата создания заказа;
- `RequiredDate` - желаемая дата доставки;
- `ShippedDate` - фактическая дата отправки;
- `ShipVia` - служба доставки;
- `Freight` - стоимость доставки;
- `ShipCity` - город доставки;
- `ShipCountry` - страна доставки.

Связи:

- `Orders.CustomerID` связан с `Customers.CustomerID`;
- `Orders.EmployeeID` связан с `Employees.EmployeeID`;
- `Orders.ShipVia` связан с `Shippers.ShipperID`;
- `Orders.OrderID` связан с `Order Details.OrderID`.

QA-смысл:

- заказ не должен ссылаться на несуществующего клиента;
- заказ не должен ссылаться на несуществующего сотрудника;
- заказ не должен ссылаться на несуществующую службу доставки;
- дата отправки не должна быть раньше даты заказа;
- заказ должен содержать хотя бы один товар.

### Order Details

Таблица хранит состав заказа: какие товары и в каком количестве были добавлены в заказ.

Ключевые поля:

- `OrderID` - идентификатор заказа;
- `ProductID` - идентификатор товара;
- `UnitPrice` - цена товара на момент заказа;
- `Quantity` - количество товара;
- `Discount` - скидка.

Связи:

- `Order Details.OrderID` связан с `Orders.OrderID`;
- `Order Details.ProductID` связан с `Products.ProductID`.

QA-смысл:

- каждая строка состава заказа должна ссылаться на существующий заказ;
- каждый товар в заказе должен существовать в таблице `Products`;
- количество товара должно быть больше нуля;
- цена товара должна быть больше нуля;
- скидка не должна быть отрицательной или больше 100%.

### Products

Таблица хранит товары компании.

Ключевые поля:

- `ProductID` - уникальный идентификатор товара;
- `ProductName` - название товара;
- `SupplierID` - поставщик товара;
- `CategoryID` - категория товара;
- `QuantityPerUnit` - количество товара в упаковке;
- `UnitPrice` - цена товара;
- `UnitsInStock` - количество на складе;
- `UnitsOnOrder` - количество в заказе у поставщика;
- `ReorderLevel` - минимальный уровень остатка;
- `Discontinued` - признак снятия товара с продажи.

Связи:

- `Products.SupplierID` связан с `Suppliers.SupplierID`;
- `Products.CategoryID` связан с `Categories.CategoryID`;
- `Products.ProductID` связан с `Order Details.ProductID`.

QA-смысл:

- товар должен быть связан с существующим поставщиком;
- товар должен быть связан с существующей категорией;
- цена товара не должна быть отрицательной или равной нулю;
- остатки товара не должны быть отрицательными;
- товары без заказов могут быть объектом дополнительного анализа.

### Categories

Таблица хранит категории товаров.

Ключевые поля:

- `CategoryID` - уникальный идентификатор категории;
- `CategoryName` - название категории;
- `Description` - описание категории.

Связи:

- `Categories.CategoryID` связан с `Products.CategoryID`.

QA-смысл:

- каждая категория может содержать один или несколько товаров;
- товары не должны ссылаться на несуществующую категорию;
- категории без товаров могут быть объектом анализа.

### Suppliers

Таблица хранит информацию о поставщиках товаров.

Ключевые поля:

- `SupplierID` - уникальный идентификатор поставщика;
- `CompanyName` - название компании поставщика;
- `ContactName` - имя контактного лица;
- `ContactTitle` - должность контактного лица;
- `City` - город;
- `Country` - страна;
- `Phone` - телефон.

Связи:

- `Suppliers.SupplierID` связан с `Products.SupplierID`.

QA-смысл:

- каждый товар должен быть связан с существующим поставщиком;
- поставщики без товаров могут быть объектом анализа;
- критичные контактные данные поставщика не должны быть пустыми.

### Employees

Таблица хранит информацию о сотрудниках компании.

Ключевые поля:

- `EmployeeID` - уникальный идентификатор сотрудника;
- `LastName` - фамилия;
- `FirstName` - имя;
- `Title` - должность;
- `BirthDate` - дата рождения;
- `HireDate` - дата приема на работу;
- `City` - город;
- `Country` - страна;
- `ReportsTo` - руководитель сотрудника.

Связи:

- `Employees.EmployeeID` связан с `Orders.EmployeeID`;
- `Employees.ReportsTo` связан с `Employees.EmployeeID`.

QA-смысл:

- каждый заказ должен быть связан с существующим сотрудником;
- дата приема на работу не должна быть раньше даты рождения;
- сотрудник не должен ссылаться сам на себя как на руководителя.

### Shippers

Таблица хранит службы доставки.

Ключевые поля:

- `ShipperID` - уникальный идентификатор службы доставки;
- `CompanyName` - название службы доставки;
- `Phone` - телефон.

Связи:

- `Shippers.ShipperID` связан с `Orders.ShipVia`.

QA-смысл:

- каждый доставленный заказ должен быть связан с существующей службой доставки;
- служба доставки без заказов может быть объектом анализа.

## Основные связи между таблицами

```text
Customers.CustomerID      -> Orders.CustomerID
Employees.EmployeeID      -> Orders.EmployeeID
Shippers.ShipperID        -> Orders.ShipVia
Orders.OrderID            -> Order Details.OrderID
Products.ProductID        -> Order Details.ProductID
Categories.CategoryID     -> Products.CategoryID
Suppliers.SupplierID      -> Products.SupplierID
Employees.EmployeeID      -> Employees.ReportsTo
```

## Проверки, которые следуют из схемы

На основе схемы базы можно подготовить SQL-проверки:

- найти заказы без существующего клиента;
- найти заказы без существующего сотрудника;
- найти заказы без существующей службы доставки;
- найти строки `Order Details` без существующего заказа;
- найти строки `Order Details` без существующего товара;
- найти товары без существующей категории;
- найти товары без существующего поставщика;
- найти заказы без товаров;
- найти товары с нулевой или отрицательной ценой;
- найти строки заказа с нулевым или отрицательным количеством;
- найти скидки меньше 0 или больше 1;
- найти даты отправки раньше даты заказа;
- найти сотрудников, у которых дата приема на работу раньше даты рождения;
- найти сотрудников, которые указаны сами себе как руководитель.
