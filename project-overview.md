# Project Overview

Этот проект посвящен SQL-тестированию учебной базы данных Northwind.

Northwind моделирует работу торговой компании, которая продает товары клиентам, оформляет заказы, взаимодействует с поставщиками и передает заказы службам доставки.

База данных используется как тестовый объект для проверки качества данных, связей между таблицами и базовой бизнес-логики.

## Тестируемый объект

Тестируемый объект - база данных Northwind SQLite:

[jpwhite3/northwind-SQLite3](https://github.com/jpwhite3/northwind-SQLite3)

База содержит готовые таблицы и тестовые данные, поэтому ее можно использовать для практики SQL-запросов и QA-проверок без разработки отдельного приложения.

## Предметная область

Northwind описывает процессы торговой компании:

- хранение информации о клиентах;
- хранение информации о товарах;
- распределение товаров по категориям;
- работа с поставщиками;
- оформление заказов;
- добавление товаров в заказ;
- назначение ответственного сотрудника;
- передача заказа в службу доставки.

## Цель тестирования

Цель тестирования - проверить данные в базе Northwind с точки зрения QA:

- корректны ли связи между таблицами;
- нет ли заказов без клиентов;
- нет ли заказов без товаров;
- существуют ли товары, указанные в заказах;
- корректно ли заполнены данные клиентов и поставщиков;
- нет ли некорректных цен, количеств и скидок;
- логичны ли даты заказа, отправки и доставки;
- можно ли найти подозрительные или потенциально ошибочные данные.

## Границы тестирования

В рамках проекта тестируется только база данных и данные внутри нее.

В проект не входит:

- тестирование UI;
- тестирование API;
- тестирование производительности;
- тестирование безопасности;
- изменение исходной базы данных;
- разработка отдельного приложения поверх базы.

## Подход к тестированию

Проверки выполняются с помощью SQL-запросов.

Основной подход:

- изучить структуру базы данных;
- определить ключевые таблицы и связи;
- составить чек-лист SQL-проверок;
- написать SQL-запросы для проверки данных;
- зафиксировать найденные проблемы как баг-репорты;
- подготовить итоговый тест-отчет.

## Основные риски данных

В проекте будут проверяться следующие риски:

- нарушенные связи между таблицами;
- отсутствующие обязательные данные;
- дублирующиеся или подозрительные записи;
- некорректные числовые значения;
- нелогичные даты;
- товары без категорий или поставщиков;
- заказы без товаров;
- клиенты без заказов;
- товары, которые не используются в заказах.

## Ожидаемый результат

В результате проекта будет подготовлен набор SQL-проверок и QA-документации, который показывает умение анализировать базу данных и использовать SQL для поиска дефектов в данных.
